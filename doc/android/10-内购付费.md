## 技术方案要点

### 统一方案：RevenueCat

双端内购均通过 **[RevenueCat](https://www.revenuecat.com/)** 接入：由 RevenueCat SDK 对接底层 **Google Play Billing**（及 iOS 侧 StoreKit），在 Dashboard 配置 **Entitlements / Offerings / Products**，应用内以 **`CustomerInfo`** 判断权益，减少双端各自维护收据校验与订阅状态机的成本。

### Android SDK 与配置

- 依赖官方 **RevenueCat Android SDK**（`com.revenuecat.purchases:purchases`）；在 `Application.onCreate` 中 **`Purchases.configure`**，传入 **Google Play 应用** 在 RevenueCat 项目中的 **Public API Key**。
- 在 Google Play Console 配置应用内商品（订阅、一次性），与 RevenueCat 后台 **Product ID** 一一绑定；订阅需处理 **base plan / offer** 与 RevenueCat **Offering** 的映射关系（以 RevenueCat 文档为准）。
- 购买流程：拉取 **`Offerings`** → 用户选包 → **`purchase`** / **`purchasePackage`** → 成功后在 **`CustomerInfo`** 中读取 **Entitlement** 是否 `active`。
- **恢复购买**：调用 **`restorePurchases`**（或依赖 RevenueCat 与账号同步机制），更新本地权益。

### 权益与 Domain

- 免费版限额、高级版能力（存储上限、AI 次数、水印、伪装图标、闯入抓拍等）在 **Domain** 层根据 **Entitlement 标识**（与 RevenueCat Dashboard 命名一致）判断；避免在 UI 直接解析 `CustomerInfo` 细节，由 **SubscriptionRepository** 封装。
- 本地可缓存「上次已知权益」用于离线展示，以 RevenueCat 缓存与刷新策略为准；**仍以商店为最终权威**，产品规则需在 Dashboard 与双端常量表对齐。

### 与直接接 Billing 的关系

- **不再**在业务代码中直接使用 `BillingClient` 主流程；底层仍走 Play Billing，由 RevenueCat 维护升级与兼容性。若遇 SDK 问题，保留查阅 Play 文档排查商店侧配置。

### 隐私与合规

- RevenueCat 会处理购买相关元数据；在隐私政策与 **Google Play 数据安全** 表单中披露第三方（RevenueCat）及用途；勿向 RevenueCat 发送用户照片等敏感内容。

## 分层落位（参考）

- **Platform**：`Purchases` 初始化、购买/恢复封装、错误码映射。
- **Domain**：会员权益矩阵 ↔ Entitlement ID。
- **Data**：可选持久化「上次同步的权益快照」用于冷启动体验（非权威）。

### 一致性

- 与 iOS 使用 **同一 RevenueCat 项目**、同一套 Entitlement / Offering 命名；权益矩阵与产品文档一致。
